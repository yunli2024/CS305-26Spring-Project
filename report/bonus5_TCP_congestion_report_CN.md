### Bonus5 TCP Congestion Control Report

> 12410922

#### 1. 实验背景

 Bonus5 选择研究 **TCP congestion control behavior**，也就是 TCP 拥塞控制算法在不同网络条件下的表现。

TCP 并不是一直用固定速度发送数据。当网络出现拥塞或丢包时，TCP 会主动调整发送窗口，从而改变吞吐量。不同拥塞控制算法的调整方式不同，因此实际性能也会不同。

本实验主要比较 Linux 中两种常见 TCP 拥塞控制算法：

- `Reno`：经典 TCP 拥塞控制算法，遇到丢包后会降低发送窗口，随后线性恢复，较为缓慢。
- `CUBIC`：Linux 中常见的默认算法，在很多场景下窗口增长更积极。

实验希望观察两个问题：

- 单条 TCP 流中，Reno 和 CUBIC 的吞吐量是否不同。由于CUBIC窗口恢复更快，预计观察到更大的吞吐量。
- Reno 和 CUBIC 同时竞争同一条瓶颈链路时，带宽分配是否公平。

实验使用 Mininet 搭建如下拓扑：

节点作用如下：

| 节点 | 作用 |
| --- | --- |
| h1 | TCP 发送端，用于 Reno 测试 |
| h2 | TCP 接收端，运行 `iperf` server |
| h3 | TCP 发送端，用于 CUBIC 测试 |
| h4 | TCP 接收端，运行 `iperf` server |
| s1-s2 | 瓶颈链路 |

链路设置如下：

| 链路 | 带宽 | 延迟 | 丢包率 | 队列大小 | 说明 |
| --- | ---: | ---: | ---: | ---: | --- |
| hi-si | 100 Mbps | 1 ms | 0% | 默认 | 避免成为瓶颈 |
| s1-s2 | 10 Mbps | 30 ms | 0.5% | 100 packets | 人为制造瓶颈 |

这里的 `0.5%` loss 是通过 Mininet 的 `TCLink` 直接设置的随机丢包率。它不是完整模拟真实网络中“队列被填满后溢出丢包”的全过程，而是人为制造一种可控的丢包环境，用来观察 Reno 和 CUBIC 在遇到丢包后的恢复能力。

真实拥塞通常需要多条流同时挤占瓶颈链路，让交换机队列逐渐排满，最后发生队列溢出丢包。这个过程会受到发送时机、队列大小、调度延迟、主机性能和 Mininet 运行状态的影响，因此每次运行的结果可能差异较大，不容易稳定复现。

随机丢包和真实拥塞的区别是：

- 真实拥塞丢包来自队列溢出，通常会伴随排队延迟升高。
- 随机丢包由链路参数直接控制，不一定表示队列已经排满。
- 但对传统 TCP 来说，丢包本身就是重要的拥塞信号，因此随机 loss 仍然可以用来比较 Reno 和 CUBIC 对丢包的反应差异。

因此，本实验重点不是完整复现一个真实网络拥塞过程，而是控制其他条件不变，通过固定 loss 来比较两种拥塞控制算法在丢包后的吞吐表现和恢复能力。

实验分为两部分：

1. **单流吞吐量测试**：分别运行 `Reno` 和 `CUBIC`，比较它们在同一瓶颈链路下的 TCP throughput。
2. **公平性测试**：同时运行 `h1 -> h2` 的 Reno 流和 `h3 -> h4` 的 CUBIC 流，观察二者共享瓶颈链路时的带宽分配。

公平性使用  $\text{Jain's Fairness Index}$  衡量：

$J=\frac{\left(\sum_{i=1}^{n} x_i\right)^2}{n\sum_{i=1}^{n}x_i^2}$

在这个两条链路的场景下：

$J=\frac{(x_1+x_2)^2}{2(x_1^2+x_2^2)}$

其中 $x_1$，$x_2$ 是两条流的吞吐量。结果越接近 $1$，说明越公平。

#### 3. 测试方法

运行命令：

```bash
sudo mn -c
sudo env "PATH=$PATH" python tests/bonus5_tcp_congestion/test_tcp_congestion.py
```

脚本会自动完成：

- 使用 Reno 运行一次单流 `iperf` 测试。
- 使用 CUBIC 运行一次单流 `iperf` 测试。
- 同时运行 Reno 和 CUBIC 两条流，计算公平性。

#### 4. 测试结果

本次实验多次运行，得到的关键结果如下：

| 实验次数 | Reno 单流吞吐量 | CUBIC 单流吞吐量 | 双流 Reno 吞吐量 | 双流 CUBIC 吞吐量 | Jain fairness index |
| --- | ---: | ---: | ---: | ---: | ---: |
| 1 | 3.53 Mbps | 5.66 Mbps | 4.34 Mbps | 3.34 Mbps | 0.983 |
| 2 | 6.37 Mbps | 6.84 Mbps | 4.70 Mbps | 5.79 Mbps | 0.989 |
| 3 | 3.91 Mbps | 5.54 Mbps | 5.65 Mbps | 3.68 Mbps | 0.957 |
| avg | 4.60 Mbps | 6.01 Mbps | 4.90 Mbps | 4.27 Mbps | 0.976 |

从单流结果可以看到，本组实验中 CUBIC 的平均吞吐量为 `6.01 Mbps`，Reno 的平均吞吐量为 `4.60 Mbps`，因此 CUBIC 的整体吞吐表现更好。

**需要注意的是，TCP 实验具有一定波动性。**随机丢包、Mininet 调度、iperf 启动时机等因素都会影响单次结果，所以 CUBIC 的吞吐量不一定在每一次运行中都大于 Reno。本报告关注的是多次实验后的平均结果，而不是某一次单独运行的结果。

本次记录的三次单流实验说明在本实验记录的瓶颈链路和丢包条件下，CUBIC 的单流吞吐性能整体更好。原因是 Reno 在检测到丢包后会比较保守地降低发送窗口，而 CUBIC 的窗口恢复方式更积极，通常能更快恢复到较高发送速率。

在公平性实验中，三次 Jain fairness index 都比较接近 `1`，平均值为 `0.976`，说明 Reno 和 CUBIC 同时经过同一瓶颈链路时，整体带宽分配是比较公平的。不过每次实验中谁获得更多带宽并不完全固定，这是因为 TCP 会受到随机丢包、启动时机和 Mininet 调度的影响。

#### 5. 总结

本次 Bonus5 使用 Mininet 和 `iperf` 研究了 TCP 拥塞控制算法的行为。实验通过设置一条带宽较低、存在丢包的瓶颈链路，对比了 Reno 和 CUBIC 的吞吐量表现，并用 Jain fairness index 分析了两种算法同时竞争带宽时的公平性。

实验结果表明，在单流情况下，CUBIC 的平均吞吐量整体高于 Reno，说明 CUBIC 在本实验场景下能更好地利用瓶颈链路。但由于网络实验存在随机性，CUBIC 不一定在每一次单独运行中都高于 Reno，因此这里采用多次实验结果进行对比。在双流竞争情况下，fairness index 均接近 `1`，说明 Reno 和 CUBIC 共享瓶颈链路时整体较公平。

本实验说明，TCP 的实际传输性能不仅由链路带宽决定，也会受到拥塞控制算法、丢包率和竞争流数量的影响。
